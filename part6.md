## 2. Background Concept: Sessions & Cookies

Before understanding Sequencer, it's essential to understand **sessions** — since Sequencer is primarily used to analyze session tokens/IDs.

### 2.1 Why HTTP is Stateless

- **HTTP is a stateless protocol.** This means that when a request goes from a user (client) to a server, the server **has no inherent memory** of where that request came from or what happened before it.
- Once a single request-response cycle completes, that "session" (in the raw HTTP sense) effectively ends — the server doesn't automatically remember the client for the next request.

### 2.2 How Sessions Work: Cookies & Session IDs

To work around HTTP's statelessness, servers **maintain sessions** using **cookies** installed on the client's system/browser.

- **Cookies** can store: user information, user preferences, and — importantly — a **Session ID**.
- When you log in to a website, the server issues you a **Session ID**, stored in a cookie on your browser.
- This Session ID is used by the server to verify: _"Is this the same legitimate user who logged in, or could this be a different/malicious user pretending to be them?"_

**Key distinction:**

- **Session ID / Token** → generated and maintained on the **server** side.
- **Cookie** → stored on the **client's (user's) browser**, and typically contains the session ID (among other data).

**Why Sequencer matters:** Sequencer is used when performing **session fixation attacks** or **session management testing** — to check **how** a target application generates its session IDs/tokens. This matters because:

> If a server generates **weak, predictable session IDs**, an attacker could potentially **guess** a valid session ID and log into someone else's account — **without ever needing their actual username or password.**

### 3.1 The School ID Card Analogy

Imagine you join a school and are issued an ID card with a number — say, `1234`. If another student's ID number is `12345`, then a third student's number might reasonably be guessed as `12346`. If ID numbers follow an easily predictable pattern, anyone could **guess** another student's number.

The same risk applies between a server and client: if the server generates session IDs in a predictable pattern, those IDs can be **guessed** by an attacker. A properly secure server must generate:

- **Random** session IDs, and
- **Unpredictable** session IDs

**Sequencer's job:** To test and measure exactly _how_ random/unpredictable a target application's session IDs actually are.

## 3. What is the Sequencer Module?

**Sequencer** is a Burp Suite tool used to **statistically analyze the randomness (entropy) of session tokens or other supposedly-random values** generated by a web application (e.g., session IDs, CSRF tokens, password reset tokens).

It works by collecting a **large sample** of tokens from the target application and running statistical tests to determine how predictable or random those tokens really are.

## 4. Practical: Using Sequencer

### 4.1 Step-by-Step Workflow

1. Navigate to the practice target (e.g., **bWAPP**) login page.
2. Go to the **Proxy** tab and turn **Intercept ON**.
3. Enter login credentials (e.g., `bee` / `bug`) on the target site and submit.
4. The login request is intercepted in Burp.
5. **Right-click** the intercepted request → **Send to Sequencer**.
6. In the **Sequencer** tab, the relevant token (e.g., an **HTTP session ID / PHP session ID**, since session IDs can be generated differently across platforms) is usually **auto-selected**.
   - If you need to analyze a different parameter manually (e.g., a specific entry, an email ID, etc.), you can use the **Custom Location** option to manually select exactly which value to analyze.
7. Select the appropriate source (e.g., **Cookie**) for the token being analyzed.
8. Click **Start Live Capture**.
9. Burp will now send **multiple repeated requests** to the target system, collecting a fresh session token from each response.
   - This process typically needs **several hundred to several thousand samples** (e.g., 500–1000, sometimes up to 10,000+) to produce a statistically meaningful analysis — but for testing/demo purposes, a smaller sample (e.g., ~500–1000) is often sufficient.
10. Once you have enough samples, click **Stop**, then click **Analyze Now**.

### 4.2 Understanding the Analysis Report

After clicking **Analyze Now**, Burp generates a detailed report, including:

- **Overall Result:** A summary verdict on the token's quality.
  - Example finding: _"Effective entropy is estimated to be [X] bits and estimated to be Excellent."_
  - **"Excellent"** here means the tokens are being generated with a **very high degree of randomness** — not something that could be easily guessed.
- A **chart** showing **"Number of Bits of Entropy"** against a **significance level**.

### 4.3 Understanding Entropy & Bits

**Entropy** is the standard measure used to quantify the **randomness** of session IDs/tokens — the higher the entropy, the harder the token is to predict/guess.

**Industry/cryptographic standard:** Modern applications should aim for **at least 128 bits** of entropy in their session tokens.

**What does "128 bits" actually mean?**

- It refers to the number of possible combinations: **2^128** (2 multiplied by itself 128 times).
- **Simple example:** 2² = 2 × 2 = 4 → this means there would only be **4 possible combinations** to guess from.
- Scaling this up: **2^128** represents an astronomically large number of possible combinations that an attacker would need to try in order to correctly guess a valid session ID.
- **The higher the number of bits → the more possible combinations → the harder the token is to brute-force/guess → the stronger and more secure the session ID.**

> Note: In a real demonstration, a smaller sample size (analysis stopped early) might show a lower estimated entropy (e.g., ~106 bits) purely due to limited sample size — but with a full, proper sample (e.g., ~1000+ requests), most modern, well-built applications will typically show entropy at or above the ~128-bit benchmark.

**Bit-level analysis:** Sequencer's report also allows drilling down into **bit-level analysis** for a more granular statistical view, useful for advanced researchers (bug bounty hunters, penetration testers) studying session management or session fixation vulnerabilities in depth.

## 5. Viewing and Testing Tokens Directly

To see what actual session tokens look like:

1. Go to a saved sample of collected tokens (e.g., a "Tokens" tab/file within Sequencer).
2. You'll see the **raw, randomly-generated tokens** that were captured during the live capture process.
   - These need to be sufficiently **random** — if they weren't, an attacker could guess one and log in as another user **without ever entering a valid username/password**.

**Demonstrating token changes over time:**

1. Right-click on a captured token/request → **Send to Repeater**.
2. Click **Send** in Repeater.
3. Observe the **response** — it includes a fresh **Session ID** issued by the server.
4. Click **Send** again — notice that **the session ID changes every single time**.
   - This is intentional: session IDs are deliberately rotated/changed frequently so that an attacker **cannot "fix" or predict** a session ID and hijack it later (this is the core defense against **session fixation attacks**).

## 6. What is the Decoder Module?

**Decoder** is used when a captured request/response contains data in an **encoded format that isn't directly human-readable**, and you need to either:

- **Decode** it (convert it back into readable plain text), or
- **Encode** it (convert plain text into a specific format)

**Example demonstrated:**

- Take a plain text value (e.g., a name).
- Decoder offers multiple **encoding options** — e.g., **URL encoding**.
- Applying URL encoding converts the plain text into its URL-encoded equivalent.

**Use case:** Decoder is commonly used when working with things like **ciphers**, or various online platforms/CTF challenges where data has been obfuscated — allowing you to **encode or decode** data in multiple standard formats directly within Burp Suite.

---

## 7. What is the Comparer Module?

**Comparer** is used to **compare two requests, responses, or values** side by side, to identify exactly what has changed between them (additions, deletions, or modifications).

**Practical workflow demonstrated:**

1. Go to **Proxy**, intercept a request.
2. Right-click → **Send to Comparer**.
3. Perform another related action, intercept the resulting new request.
4. Right-click that second request → **Send to Comparer** as well.
5. Now with **two values loaded** in Comparer, click **Words** (word-level comparison).
6. Burp highlights the differences:
   - **Text highlighted in one color (e.g., red/brown)** → indicates content that was **modified/deleted** compared to the other request.
     - Example: In the first request, a field like `POST /login` or a title parameter was present, but is **deleted/missing** in the second.
   - **Text highlighted in another color (e.g., yellow)** → indicates content that is **newly added** in the compared request.

**Why Comparer is important:** When you need to analyze differences between two separate requests or responses — to spot exactly what data was **added, removed, or modified** between them — Comparer makes this quick and visual, rather than manually reading through both side by side.

## 8. The Logger Tab

- **Logger** keeps a running record of **all your logs/activity** within Burp Suite — similar in concept to how **Windows Event Logs** record system activity.
- Every request/response that passes through Burp gets logged here, giving you a complete historical record to review later.
